42 Inception

Vollständiger Guide zum 42-Inception-Projekt —
Docker, Docker Compose, NGINX, WordPress & MariaDB in einer VM.

🔵 42 Project 🟢 Docker 🏗️ Infrastructure 📅 2026
ℹ️
Was ist Inception? Du baust eine kleine Infrastruktur mit Docker Compose auf, die komplett in einer Virtual Machine läuft. Jeder Service (NGINX, WordPress, MariaDB) bekommt seinen eigenen Container mit eigenem Dockerfile. Keine fertigen Images (außer Alpine/Debian base), kein docker-compose pull.
💻 Setup

VM & Grundlagen

📁
Projektstruktur ▶

Das Subject gibt eine feste Ordnerstruktur vor. Alles liegt unter srcs/, das Makefile liegt im Root.

Projektstruktur 
inception/
├── Makefile
└── srcs/
    ├── docker-compose.yml
    ├── .env
    └── requirements/
        ├── nginx/
        │   ├── Dockerfile
        │   ├── conf/
        │   │   └── nginx.conf
        │   └── tools/           # optional: setup scripts
        ├── wordpress/
        │   ├── Dockerfile
        │   ├── conf/
        │   │   └── www.conf
        │   └── tools/
        │       └── setup.sh
        └── mariadb/
            ├── Dockerfile
            ├── conf/
            │   └── 50-server.cnf
            └── tools/
                └── setup.sh

📝 Wichtige Regeln

  • Die .env-Datei enthält alle Credentials und darf NICHT in Git committed werden
  • Jeder Service hat sein eigenes Dockerfile
  • Dockerfiles werden über docker-compose.yml aufgerufen (mit build:)
  • Nur der NGINX-Container hat Port 443 offen nach außen
🔑
.env Datei (Credentials) ▶

Alle Passwörter, Benutzernamen und Datenbank-Credentials kommen in die .env-Datei. Docker Compose liest sie automatisch ein.

bash · srcs/.env 
# Domain
DOMAIN_NAME=login.42.fr

# MariaDB
MYSQL_ROOT_PASSWORD=supersecretroot
MYSQL_DATABASE=wordpress
MYSQL_USER=wpuser
MYSQL_PASSWORD=wpuserpassword

# WordPress
WP_ADMIN_USER=boss
WP_ADMIN_PASSWORD=bosspassword
WP_ADMIN_EMAIL=boss@42.fr
WP_USER=writer
WP_USER_PASSWORD=writerpassword
WP_USER_EMAIL=writer@42.fr

⚠️ Wichtig

  • Die .env in .gitignore eintragen!
  • Der WP Admin darf NICHT "admin" oder "administrator" als Username haben
  • Passwörter sollten stark genug sein (Subject-Anforderung)
🖥️
Virtual Machine einrichten ▶

Du brauchst eine VM (z.B. mit VirtualBox oder UTM auf Mac). Debian oder Alpine als OS. Docker und Docker Compose müssen installiert werden.

bash · VM Setup (Debian) 
# System updaten
sudo apt update && sudo apt upgrade -y

# Docker installieren
sudo apt install -y docker.io docker-compose

# User zur Docker-Gruppe hinzufügen (kein sudo nötig)
sudo usermod -aG docker $USER
# Danach ausloggen und neu einloggen!

# Domain in /etc/hosts eintragen
echo "127.0.0.1 login.42.fr" | sudo tee -a /etc/hosts

# Prüfen ob Docker läuft
docker --version
docker-compose --version
🐳 Docker

Docker & Docker Compose

📦
Docker Grundkonzepte ▶

Docker isoliert Anwendungen in Containern. Jeder Container basiert auf einem Image, das durch ein Dockerfile definiert wird. Container sind leichtgewichtig, starten schnell und teilen sich den Host-Kernel.

BegriffErklärung
ImageSchreibgeschütztes Template — die "Bauanleitung" für einen Container
ContainerLaufende Instanz eines Images — isolierter Prozess
DockerfileTextdatei mit Anweisungen zum Bauen eines Images
VolumePersistenter Speicher, überlebt Container-Neustarts
NetworkVirtuelles Netzwerk für Container-zu-Container-Kommunikation
docker-composeTool um Multi-Container-Setups deklarativ zu definieren

📝 Inception-Regeln

  • Nur alpine (penultimate stable) oder debian (penultimate stable) als Base-Image
  • Keine fertigen Images von Docker Hub (kein nginx:latest, kein wordpress:latest)
  • Jeder Service braucht ein eigenes Dockerfile
  • Container müssen bei Crash automatisch neu starten (restart: unless-stopped)
  • Keine tail -f, sleep infinity oder ähnliche Hacks
📋
docker-compose.yml ▶

Die zentrale Konfigurationsdatei. Hier werden alle Services, Volumes und Networks definiert. Docker Compose baut und startet alles mit einem Befehl.

yaml · srcs/docker-compose.yml 
version: '3.8'

services:
  nginx:
    build: ./requirements/nginx
    container_name: nginx
    ports:
      - "443:443"
    volumes:
      - wordpress_data:/var/www/html
    networks:
      - inception
    depends_on:
      - wordpress
    restart: unless-stopped

  wordpress:
    build: ./requirements/wordpress
    container_name: wordpress
    volumes:
      - wordpress_data:/var/www/html
    networks:
      - inception
    depends_on:
      - mariadb
    env_file:
      - .env
    restart: unless-stopped

  mariadb:
    build: ./requirements/mariadb
    container_name: mariadb
    volumes:
      - mariadb_data:/var/lib/mysql
    networks:
      - inception
    env_file:
      - .env
    restart: unless-stopped

volumes:
  wordpress_data:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: /home/${USER}/data/wordpress
  mariadb_data:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: /home/${USER}/data/mariadb

networks:
  inception:
    driver: bridge

📝 Erklärung

  • build: — Pfad zum Dockerfile (Docker Compose baut das Image selbst)
  • depends_on: — Startreihenfolge: MariaDB → WordPress → NGINX
  • volumes: — Daten persistent auf dem Host speichern unter /home/USER/data/
  • networks: — Alle Container im selben Netzwerk, können sich per Name ansprechen
  • env_file: — Lädt die Umgebungsvariablen aus der .env-Datei
  • Nur NGINX exponiert Port 443 nach außen!
🏗️
Dockerfile Best Practices ▶

Jedes Dockerfile baut ein Image Schritt für Schritt auf. Wichtig: Der letzte Befehl (CMD/ENTRYPOINT) muss ein Prozess sein, der im Vordergrund läuft (PID 1).

dockerfile · Allgemeine Struktur 
FROM    debian:bullseye          # Base Image (penultimate stable)

RUN     apt-get update && \       # Pakete installieren
        apt-get install -y \
        paket1 paket2 && \
        rm -rf /var/lib/apt/lists/*  # Cache aufräumen

COPY    conf/config.conf /etc/    # Konfigdateien rein kopieren
COPY    tools/setup.sh /usr/local/bin/

RUN     chmod +x /usr/local/bin/setup.sh

EXPOSE  443                       # Port dokumentieren

ENTRYPOINT ["setup.sh"]       # Startskript (PID 1)

⚠️ Häufige Fehler

  • Kein CMD ["tail", "-f", "/dev/null"] — Container muss echten Prozess laufen
  • Kein sleep infinity — dasselbe Problem
  • Immer exec vorm Hauptprozess im Skript nutzen (für korrektes Signal-Handling)
  • ENTRYPOINT für das Startskript, nicht CMD (wird nicht überschrieben)
🔒 NGINX

NGINX + TLSv1.2/1.3

📄
NGINX Dockerfile ▶

NGINX dient als Reverse Proxy und ist der einzige Einstiegspunkt (Port 443). Er leitet PHP-Anfragen an WordPress (php-fpm auf Port 9000) weiter und terminiert TLS.

dockerfile · requirements/nginx/Dockerfile 
FROM    debian:bullseye

RUN     apt-get update && apt-get install -y \
        nginx openssl && \
        rm -rf /var/lib/apt/lists/*

# Self-signed TLS Zertifikat generieren
RUN     mkdir -p /etc/nginx/ssl && \
        openssl req -x509 -nodes -days 365 \
        -newkey rsa:2048 \
        -keyout /etc/nginx/ssl/nginx.key \
        -out /etc/nginx/ssl/nginx.crt \
        -subj "/C=DE/ST=BW/L=HN/O=42/CN=login.42.fr"

COPY    conf/nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE  443

CMD     ["nginx", "-g", "daemon off;"]

📝 Erklärung

  • openssl req -x509 erzeugt ein selbst-signiertes SSL-Zertifikat
  • -nodes = kein Passwort auf dem Key
  • -subj = Subject des Zertifikats (CN muss dein login.42.fr sein)
  • daemon off; = NGINX läuft im Vordergrund (wichtig für Docker!)
⚙️
NGINX Konfiguration ▶

Die NGINX-Config muss TLSv1.2 oder TLSv1.3 erzwingen und PHP-Requests an den WordPress-Container weiterleiten.

nginx · requirements/nginx/conf/nginx.conf 
server {
    listen              443 ssl;
    server_name         login.42.fr;

    # TLS Konfiguration
    ssl_certificate     /etc/nginx/ssl/nginx.crt;
    ssl_certificate_key /etc/nginx/ssl/nginx.key;
    ssl_protocols       TLSv1.2 TLSv1.3;

    # WordPress Root
    root                /var/www/html;
    index               index.php index.html;

    # PHP-Requests an WordPress-Container weiterleiten
    location ~ \.php$ {
        fastcgi_pass    wordpress:9000;
        fastcgi_index   index.php;
        fastcgi_param   SCRIPT_FILENAME $document_root$fastcgi_script_name;
        include         fastcgi_params;
    }

    location / {
        try_files $uri $uri/ /index.php?$args;
    }
}

📝 Wichtige Punkte

  • fastcgi_pass wordpress:9000 — "wordpress" ist der Container-Name im Docker-Netzwerk
  • ssl_protocols TLSv1.2 TLSv1.3 — Subject erlaubt NUR diese Protokolle
  • root /var/www/html — gleiches Volume wie WordPress
  • Port 80 (HTTP) darf NICHT offen sein
📝 WordPress

WordPress + php-fpm

📄
WordPress Dockerfile ▶

WordPress wird mit php-fpm installiert (NICHT mit Apache!). php-fpm lauscht auf Port 9000 und verarbeitet PHP-Dateien, die NGINX dorthin weiterleitet.

dockerfile · requirements/wordpress/Dockerfile 
FROM    debian:bullseye

RUN     apt-get update && apt-get install -y \
        php7.4-fpm php7.4-mysql php7.4-curl \
        php7.4-gd php7.4-mbstring php7.4-xml \
        php7.4-zip wget curl && \
        rm -rf /var/lib/apt/lists/*

# WP-CLI installieren (WordPress über Command Line managen)
RUN     curl -O https://raw.githubusercontent.com/wp-cli/builds/gh-pages/phar/wp-cli.phar && \
        chmod +x wp-cli.phar && \
        mv wp-cli.phar /usr/local/bin/wp

# php-fpm soll auf Port 9000 hören (nicht auf Unix Socket)
COPY    conf/www.conf /etc/php/7.4/fpm/pool.d/www.conf

COPY    tools/setup.sh /usr/local/bin/setup.sh
RUN     chmod +x /usr/local/bin/setup.sh

WORKDIR /var/www/html

EXPOSE  9000

ENTRYPOINT ["setup.sh"]
🔧
WordPress Setup Script ▶

Das Setup-Skript wird als ENTRYPOINT ausgeführt: Es lädt WordPress herunter, konfiguriert es und startet php-fpm. Prüft vorher, ob WordPress bereits installiert ist (idempotent!).

bash · requirements/wordpress/tools/setup.sh 
#!/bin/bash

# Warten bis MariaDB bereit ist
while ! mysqladmin ping -h mariadb --silent 2>/dev/null; do
    echo "Warte auf MariaDB..."
    sleep 2
done

# WordPress nur herunterladen wenn noch nicht vorhanden
if [ ! -f /var/www/html/wp-config.php ]; then

    # WordPress Core herunterladen
    wp core download --allow-root

    # wp-config.php erstellen
    wp config create --allow-root \
        --dbname=$MYSQL_DATABASE \
        --dbuser=$MYSQL_USER \
        --dbpass=$MYSQL_PASSWORD \
        --dbhost=mariadb:3306

    # WordPress installieren
    wp core install --allow-root \
        --url="https://${DOMAIN_NAME}" \
        --title="Inception" \
        --admin_user=$WP_ADMIN_USER \
        --admin_password=$WP_ADMIN_PASSWORD \
        --admin_email=$WP_ADMIN_EMAIL

    # Zweiten User anlegen (kein Admin)
    wp user create --allow-root \
        $WP_USER $WP_USER_EMAIL \
        --role=author \
        --user_pass=$WP_USER_PASSWORD

fi

# php-fpm Verzeichnis erstellen und starten
mkdir -p /run/php
exec /usr/sbin/php-fpm7.4 -F   # -F = Vordergrund (PID 1)

📝 Wichtige Details

  • exec ersetzt die Shell durch php-fpm → korrektes Signal-Handling (PID 1)
  • Die while-Schleife wartet auf MariaDB (race condition vermeiden)
  • Das if-Statement macht das Skript idempotent (wiederholbar)
  • --allow-root ist nötig, weil WP-CLI nicht als root laufen will
  • --dbhost=mariadb:3306 — Docker DNS löst den Container-Namen auf
⚙️
php-fpm Pool Konfiguration (www.conf) ▶

Standardmäßig lauscht php-fpm auf einem Unix-Socket. Für Docker muss es auf einem TCP-Port lauschen, damit NGINX aus einem anderen Container darauf zugreifen kann.

ini · requirements/wordpress/conf/www.conf 
[www]
user = www-data
group = www-data

; Auf TCP Port 9000 lauschen statt Unix Socket
listen = 9000

pm = dynamic
pm.max_children = 5
pm.start_servers = 2
pm.min_spare_servers = 1
pm.max_spare_servers = 3
🗄️ MariaDB

MariaDB

📄
MariaDB Dockerfile ▶

MariaDB ist die Datenbank für WordPress. Der Container muss die Datenbank und den User beim ersten Start erstellen.

dockerfile · requirements/mariadb/Dockerfile 
FROM    debian:bullseye

RUN     apt-get update && apt-get install -y \
        mariadb-server mariadb-client && \
        rm -rf /var/lib/apt/lists/*

# MariaDB soll auf allen Interfaces lauschen (nicht nur localhost)
COPY    conf/50-server.cnf /etc/mysql/mariadb.conf.d/50-server.cnf

COPY    tools/setup.sh /usr/local/bin/setup.sh
RUN     chmod +x /usr/local/bin/setup.sh

EXPOSE  3306

ENTRYPOINT ["setup.sh"]
🔧
MariaDB Setup Script ▶

Das Setup-Skript initialisiert die Datenbank, erstellt den WordPress-User und setzt Berechtigungen. Danach startet es MariaDB im Vordergrund.

bash · requirements/mariadb/tools/setup.sh 
#!/bin/bash

# Datenbank-Verzeichnis initialisieren falls nötig
if [ ! -d /var/lib/mysql/${MYSQL_DATABASE} ]; then

    # MariaDB temporär starten
    service mariadb start

    # Datenbank und User erstellen
    mysql -e "CREATE DATABASE IF NOT EXISTS ${MYSQL_DATABASE};"
    mysql -e "CREATE USER IF NOT EXISTS '${MYSQL_USER}'@'%' IDENTIFIED BY '${MYSQL_PASSWORD}';"
    mysql -e "GRANT ALL PRIVILEGES ON ${MYSQL_DATABASE}.* TO '${MYSQL_USER}'@'%';"
    mysql -e "ALTER USER 'root'@'localhost' IDENTIFIED BY '${MYSQL_ROOT_PASSWORD}';"
    mysql -e "FLUSH PRIVILEGES;"

    # MariaDB stoppen (wird gleich sauber neu gestartet)
    service mariadb stop

fi

# MariaDB im Vordergrund starten (PID 1)
exec mysqld_safe

📝 Erklärung

  • '${MYSQL_USER}'@'%' — Der % erlaubt Verbindungen von jedem Host (= anderen Containern)
  • Das Root-Passwort wird ebenfalls gesetzt (Subject-Anforderung)
  • exec mysqld_safe — startet MariaDB als PID 1 im Vordergrund
  • Das if-Statement verhindert doppelte Initialisierung bei Restart
⚙️
MariaDB Konfiguration (50-server.cnf) ▶

Die wichtigste Änderung: bind-address von 127.0.0.1 auf 0.0.0.0 ändern, damit andere Container sich verbinden können.

ini · requirements/mariadb/conf/50-server.cnf 
[mysqld]
datadir         = /var/lib/mysql
socket          = /run/mysqld/mysqld.sock
bind-address    = 0.0.0.0     # Auf allen Interfaces lauschen
port            = 3306
user            = mysql
pid-file        = /run/mysqld/mysqld.pid
🌐 Infra

Volumes & Netzwerk

💾
Volumes (Persistenter Speicher) ▶

Volumes sorgen dafür, dass Daten Container-Neustarts überleben. Das Subject verlangt, dass die Daten unter /home/USER/data/ auf dem Host gespeichert werden.

Datenfluss 
# WordPress Volume:
Host:      /home/user/data/wordpress/
                    ↕ (bind mount)
Container: /var/www/html/              ← NGINX + WordPress teilen sich das!

# MariaDB Volume:
Host:      /home/user/data/mariadb/
                    ↕ (bind mount)
Container: /var/lib/mysql/

📝 Wichtig

  • Die Verzeichnisse auf dem Host müssen vor dem Start existieren (im Makefile erstellen)
  • WordPress-Dateien und MariaDB-Daten bleiben erhalten, auch wenn Container neu gebaut werden
  • NGINX und WordPress teilen sich das WordPress-Volume (NGINX braucht die PHP-Dateien)
🔗
Docker Netzwerk ▶

Alle drei Container sind im selben Docker-Netzwerk (inception). Sie können sich gegenseitig per Container-Name ansprechen — Docker DNS löst die Namen automatisch auf.

Netzwerk-Architektur 
                    ┌─────────────────────────────────────┐
                    │       Docker Network: inception       │
Browser             │                                     │
   │                │  ┌─────────┐    ┌───────────┐       │
   │  HTTPS :443    │  │  NGINX  │───▶│ WordPress │       │
   └───────────────▶│  │  :443   │    │ php-fpm   │       │
                    │  └─────────┘    │  :9000    │       │
                    │                 └─────┬─────┘       │
                    │                       │             │
                    │                 ┌─────▼─────┐       │
                    │                 │  MariaDB  │       │
                    │                 │   :3306   │       │
                    │                 └───────────┘       │
                    └─────────────────────────────────────┘

# Nur Port 443 ist von außen erreichbar!
# Interne Ports (9000, 3306) sind nur im Docker-Netzwerk sichtbar.

📝 DNS-Auflösung in Docker

  • wordpress:9000 — NGINX findet WordPress über den Container-Namen
  • mariadb:3306 — WordPress findet MariaDB über den Container-Namen
  • Kein network: host erlaubt — Container müssen isoliert sein
  • links: ist veraltet, networks: ist der richtige Weg
🛠️ Tools

Makefile & Nützliche Commands

📋
Makefile ▶

Das Makefile im Projekt-Root steuert den gesamten Lifecycle. Es erstellt die Volume-Verzeichnisse, baut die Container und räumt auf.

makefile · Makefile 
DATA_DIR = /home/$(USER)/data

all: setup
	docker-compose -f srcs/docker-compose.yml up -d --build

setup:
	mkdir -p $(DATA_DIR)/wordpress
	mkdir -p $(DATA_DIR)/mariadb

down:
	docker-compose -f srcs/docker-compose.yml down

clean: down
	docker system prune -af
	sudo rm -rf $(DATA_DIR)

re: clean all

.PHONY: all setup down clean re

📝 Make Targets

  • make / make all — Verzeichnisse erstellen, Container bauen und starten
  • make down — Container stoppen (Daten bleiben erhalten)
  • make clean — Alles entfernen inkl. Volumes und Images
  • make re — Alles neu aufbauen (clean + all)
🐛
Debugging & Nützliche Commands ▶

Diese Befehle helfen dir beim Debuggen, wenn etwas nicht funktioniert.

bash · Docker Debugging 
# Container-Status prüfen (laufen alle?)
docker ps -a

# Logs eines Containers anschauen
docker logs nginx
docker logs wordpress
docker logs mariadb

# In einen laufenden Container rein (Shell öffnen)
docker exec -it nginx bash
docker exec -it wordpress bash
docker exec -it mariadb bash

# Netzwerk prüfen (können Container sich erreichen?)
docker network ls
docker network inspect inception_inception

# Aus dem WordPress-Container MariaDB pingen
docker exec -it wordpress ping mariadb

# MySQL-Verbindung testen
docker exec -it mariadb mysql -u wpuser -p wordpress

# TLS-Zertifikat prüfen
openssl s_client -connect login.42.fr:443

# Volumes anzeigen
docker volume ls

# Alles nuklear aufräumen (Vorsicht!)
docker system prune -af --volumes

Häufige Fehler & Lösungen ▶
ProblemUrsacheLösung
502 Bad Gateway NGINX kann WordPress nicht erreichen fastcgi_pass wordpress:9000 prüfen, php-fpm läuft?
WordPress DB-Error MariaDB noch nicht bereit oder falsche Credentials Wait-Loop in setup.sh, .env-Variablen prüfen
Container startet und stoppt sofort Kein Prozess im Vordergrund (PID 1) exec + Vordergrund-Flag (-F, daemon off)
SSL-Fehler im Browser Self-signed Cert oder falscher CN CN muss login.42.fr sein, TLSv1.2+ erzwingen
