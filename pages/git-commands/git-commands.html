Git Commands

Die wichtigsten Git-Befehle — mit Erklärung und Beispielen.
Von Basics bis zu fortgeschrittenen Workflows.

Referenz Git Version Control
📦 Basics

Basics — Setup & Staging

🚀
init & clone — Repository anlegen ▶

Neues Repository erstellen oder ein bestehendes klonen. 

# Neues Repo im aktuellen Ordner initialisieren
git init

# Repo aus einer URL klonen
git clone https://github.com/user/repo.git

# In eigenen Ordner klonen
git clone https://github.com/user/repo.git mein-ordner

# Nur neuesten Commit klonen (shallow clone — schneller)
git clone --depth 1 https://github.com/user/repo.git
✅
add & commit — Änderungen speichern ▶

Dateien zum Staging-Bereich hinzufügen und committen. 

# Einzelne Datei stagen
git add datei.txt

# Alle geänderten Dateien stagen
git add .

# Interaktiv einzelne Hunks stagen
git add -p

# Commit mit Nachricht
git commit -m "feat: neues Feature hinzugefügt"

# Stagen + Committen in einem Schritt (nur bereits getrackte Dateien)
git commit -am "fix: bug behoben"
🔎
status & diff — Änderungen einsehen ▶

Aktuellen Stand des Repos und konkrete Änderungen anzeigen. 

# Überblick über geänderte Dateien
git status

# Alle unstaged Änderungen zeigen
git diff

# Staged Änderungen zeigen (was im nächsten Commit landet)
git diff --staged

# Unterschied zu einem bestimmten Commit
git diff abc1234
BefehlZeigt
git statusGeänderte, gestagete, ungetrackte Dateien
git diffUnstaged Änderungen (Arbeitsverzeichnis vs. Index)
git diff --stagedStaged Änderungen (Index vs. letzter Commit)
git diff HEADAlle Änderungen seit letztem Commit
🌿 Branches

Branching & Merging

🌿
branch & checkout — Branches verwalten ▶

Branches erstellen, wechseln und löschen. 

# Alle lokalen Branches anzeigen
git branch

# Neuen Branch erstellen
git branch feature/login

# Branch wechseln
git checkout feature/login

# Branch erstellen und direkt wechseln (kurzform)
git checkout -b feature/login

# Moderne Alternative mit switch
git switch -c feature/login

# Branch umbenennen
git branch -m alter-name neuer-name

# Lokalen Branch löschen (nur wenn gemergt)
git branch -d feature/login

# Branch erzwungen löschen
git branch -D feature/login
🔀
merge & rebase — Branches zusammenführen ▶

Merge fügt Branches zusammen und behält die History. Rebase schreibt die Commit-History neu — saubere lineare History, aber nie auf public Branches anwenden. 

# feature/login in main mergen (erst in main wechseln)
git checkout main
git merge feature/login

# Merge ohne Fast-Forward (immer einen Merge-Commit erstellen)
git merge --no-ff feature/login

# Rebase: Commits von feature auf main aufsetzen
git checkout feature/login
git rebase main

# Interaktiver Rebase: letzte 3 Commits bearbeiten
git rebase -i HEAD~3

⚠️ Merge vs. Rebase

  • Merge — History bleibt erhalten, Merge-Commit entsteht. Für Feature-Branches empfohlen.
  • Rebase — Lineare History, keine Merge-Commits. Nur auf lokalen / privaten Branches anwenden.
  • Nie git rebase auf Branches anwenden, die andere bereits gepullt haben.
☁️ Remote

Remote — Push, Pull & Fetch

🔗
remote — Remote-Verbindungen ▶

Remote-Repos verwalten (GitHub, GitLab, etc.). 

# Alle remotes anzeigen
git remote -v

# Remote hinzufügen
git remote add origin https://github.com/user/repo.git

# Remote-URL ändern
git remote set-url origin https://github.com/user/neues-repo.git

# Remote entfernen
git remote remove origin
⬆️
push & pull — Synchronisieren ▶

Commits hochladen und Änderungen herunterladen. 

# Aktuellen Branch pushen
git push

# Branch zum ersten Mal pushen + Upstream setzen
git push -u origin feature/login

# Änderungen von remote holen und mergen
git pull

# Holen + Rebase statt Merge (sauberere History)
git pull --rebase

# Nur holen, nicht mergen (fetch)
git fetch origin

# Remote Branch löschen
git push origin --delete feature/login
BefehlWas passiert
git fetchHolt Änderungen, merged NICHT — nur lokale Remote-Tracking-Refs updaten
git pullfetch + merge in einem Schritt
git pull --rebasefetch + rebase — lineare History
📜 History

History & Log

📜
log & show — Commit-History ▶

Commit-Historie durchsuchen und einzelne Commits inspizieren. 

# Commit-History anzeigen
git log

# Kompakt: eine Zeile pro Commit
git log --oneline

# Mit Branch-Graph
git log --oneline --graph --all

# Letzte 5 Commits
git log -5

# Commits eines bestimmten Autors
git log --author="Simon"

# Commits die eine Datei betreffen
git log -- datei.txt

# Details eines einzelnen Commits anzeigen
git show abc1234

# Wer hat welche Zeile zuletzt geändert?
git blame datei.txt
↩️ Undoing

Undoing — Änderungen rückgängig machen

↩️
revert, reset & restore ▶

Commits oder Änderungen rückgängig machen — je nach Situation die richtige Methode wählen. 

# Einzelne Datei auf letzten Commit zurücksetzen
git restore datei.txt

# Datei aus Staging entfernen (unstage)
git restore --staged datei.txt

# Neuen Commit erstellen der einen alten Commit rückgängig macht (sicher)
git revert abc1234

# Letzten Commit rückgängig — Änderungen bleiben im Arbeitsverzeichnis
git reset HEAD~1

# Letzten Commit und seine Änderungen komplett löschen (destruktiv!)
git reset --hard HEAD~1

# Letzten Commit-Message ändern (nur wenn noch nicht gepusht)
git commit --amend -m "korrigierte Nachricht"

⚠️ reset --hard ist destruktiv

  • git revert — sicher, erstellt neuen Commit. Für pushed Commits.
  • git reset — Commits aus History entfernen. Nur für lokale, ungepushte Commits.
  • git reset --hard — löscht Änderungen unwiderruflich. Backup vorher!
BefehlStagingArbeitsverzeichnisHistory
reset --soft HEAD~1behält stagedunverändertCommit weg
reset HEAD~1unstagedÄnderungen bleibenCommit weg
reset --hard HEAD~1geleertÄnderungen wegCommit weg
📦 Stash

Stash — Änderungen zwischenspeichern

📦
stash — Temporär weglegen ▶

Unfertige Änderungen temporär wegpacken, um den Branch zu wechseln oder etwas anderes zu erledigen. 

# Aktuelle Änderungen stashen
git stash

# Mit Beschreibung stashen
git stash push -m "WIP: login-formular"

# Alle Stashes anzeigen
git stash list

# Letzten Stash wiederherstellen (und aus Liste entfernen)
git stash pop

# Bestimmten Stash wiederherstellen
git stash apply stash@{2}

# Stash löschen
git stash drop stash@{0}

# Alle Stashes löschen
git stash clear
⚡ Advanced

Advanced — Fortgeschrittene Befehle

🍒
cherry-pick — Einzelne Commits übernehmen ▶

Einen einzelnen Commit aus einem anderen Branch in den aktuellen Branch übernehmen. 

# Einzelnen Commit in aktuellen Branch holen
git cherry-pick abc1234

# Mehrere Commits (Range)
git cherry-pick abc1234..def5678

# Cherry-pick ohne direkt zu committen (nur stagen)
git cherry-pick -n abc1234
🏷️
tag — Versionen markieren ▶

Commits mit einem Tag markieren — typisch für Releases (v1.0.0, v2.3.1). 

# Alle Tags anzeigen
git tag

# Lightweight Tag erstellen
git tag v1.0.0

# Annotated Tag mit Nachricht (empfohlen für Releases)
git tag -a v1.0.0 -m "Release v1.0.0"

# Tags pushen
git push origin --tags

# Tag löschen (lokal + remote)
git tag -d v1.0.0
git push origin --delete v1.0.0
⚙️
config — Git konfigurieren ▶

Git global oder pro-Repo konfigurieren. 

# Name und E-Mail setzen (global)
git config --global user.name "Simon Wied"
git config --global user.email "simon@example.com"

# Standard-Editor setzen
git config --global core.editor "code --wait"

# Alle globalen Einstellungen anzeigen
git config --global --list

# Aliases definieren
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.lg "log --oneline --graph --all"
AliasKurzform für
git stgit status
git cogit checkout
git lggit log --oneline --graph --all
🏢 Use Cases

Use Cases — Professionelle Workflows

🌿
Feature Branch Workflow ▶

Standard-Workflow in fast jedem Team: jedes Feature bekommt einen eigenen Branch. main bleibt immer stabil und deploybar.

01.Von main aus neuen Feature-Branch erstellen.
02.Feature entwickeln, regelmäßig committen.
03.Pull Request öffnen, Code Review abwarten.
04.Nach Approval in main mergen.
05.Feature-Branch löschen.
# 1. Aktuellen main holen
git checkout main
git pull

# 2. Feature-Branch erstellen
git checkout -b feature/user-auth

# 3. Entwickeln & committen
git add .
git commit -m "feat: add JWT authentication"

# 4. Branch pushen + Pull Request öffnen
git push -u origin feature/user-auth

# 5. Nach Merge: aufräumen
git checkout main
git pull
git branch -d feature/user-auth
🚨
Hotfix — Produktionsbug schnell fixen ▶

Kritischer Bug in Produktion — direkt vom main-Branch aus fixen, ohne laufende Features zu stören.

01.Laufende Arbeit stashen.
02.Hotfix-Branch direkt von main anlegen.
03.Fix committen, in main mergen.
04.Stash zurück, weiterarbeiten.
# 1. Laufende Arbeit sichern
git stash push -m "WIP: neue-feature"

# 2. Hotfix-Branch von main
git checkout main
git pull
git checkout -b hotfix/login-crash

# 3. Fix committen
git commit -am "fix: prevent null pointer on login"

# 4. In main mergen + pushen
git checkout main
git merge --no-ff hotfix/login-crash
git push
git tag -a v1.2.1 -m "Hotfix: login crash"
git push origin --tags

# 5. Weiterarbeiten
git checkout feature/neue-feature
git stash pop
🧹
Branch aufräumen vor Pull Request ▶

Vor einem PR die Commits squashen (zu einem zusammenfassen) und den Branch auf den aktuellen main rebasen — saubere History für Reviewer.

01.main pullen.
02.Feature-Branch auf aktuellen main rebasen.
03.Commits interaktiv squashen.
04.Branch force-pushen (da History neu geschrieben).
# 1. main aktualisieren
git checkout main && git pull

# 2. Feature-Branch auf main rebasen
git checkout feature/user-auth
git rebase main

# 3. Letzte 4 Commits zu einem zusammenfassen
git rebase -i HEAD~4
# Im Editor: pick für ersten, squash (s) für alle weiteren

# 4. Force-push (nötig weil History neu geschrieben)
git push --force-with-lease

💡 --force-with-lease statt --force

  • --force überschreibt blind — gefährlich wenn jemand anderes gepusht hat.
  • --force-with-lease bricht ab wenn remote neuer ist als erwartet. Immer bevorzugen.
🔄
Code Review Feedback einarbeiten ▶

PR ist offen, Reviewer haben Feedback gegeben — Änderungen in denselben Branch committen und pushen. 

# Änderungen nach Review committen
git add .
git commit -m "refactor: address review comments"
git push

# Wenn man letzten Commit erweitern will (noch nicht gepusht)
git add .
git commit --amend --no-edit
git push --force-with-lease

# Wenn main sich inzwischen geändert hat: rebasen
git fetch origin
git rebase origin/main
git push --force-with-lease
🚀
Release vorbereiten & taggen ▶

Neues Release ausliefern — Changelog aktualisieren, Version taggen, CI/CD triggert den Deploy.

01.main ist stabil und alle PRs gemergt.
02.Version in package.json / Changelog anpassen.
03.Commit + annotated Tag erstellen.
04.Tag pushen — löst Pipeline aus.
# 1. Sicherstellen, dass main aktuell ist
git checkout main
git pull

# 2. Version-Commit
git commit -am "chore: bump version to 2.1.0"

# 3. Annotated Tag
git tag -a v2.1.0 -m "Release v2.1.0 — neue Dashboard-Features"

# 4. Pushen (Commit + Tag)
git push
git push origin v2.1.0
⚔️
Merge Conflict lösen ▶

Zwei Branches haben dieselbe Datei geändert — Git kann nicht automatisch mergen. Konflikt manuell auflösen.

01.git merge oder rebase schlägt fehl, zeigt Konflikt-Dateien.
02.Konflikt-Markierungen in den Dateien manuell auflösen.
03.Gelöste Dateien stagen.
04.Merge / Rebase abschließen.
# Merge schlägt fehl — Konflikte anzeigen
git status
# → "both modified: src/auth.js"

# Konflikt-Datei öffnen und Markierungen suchen:
# <<<<<<< HEAD     (eigene Änderung)
# =======           (Trennlinie)
# >>>>>>> main     (incoming Änderung)

# Nach manuellem Lösen: stagen + abschließen
git add src/auth.js
git merge --continue
# oder bei rebase:
git rebase --continue

# Merge/Rebase abbrechen (zurück zum Ausgangszustand)
git merge --abort
git rebase --abort
😬
Auf falschem Branch committet ▶

Commit landet versehentlich auf dem falschen Branch (z.B. direkt auf main statt auf einem Feature-Branch). Commit auf den richtigen Branch verschieben und vom falschen entfernen.

01.Commit-Hash des falschen Commits notieren.
02.Auf den richtigen Branch wechseln.
03.Commit per cherry-pick übernehmen.
04.Zurück auf den falschen Branch, Commit entfernen.
# 1. Hash des versehentlichen Commits merken
git log --oneline -5
# → abc1234 FEAT: neues Feature  ← der falsche Commit

# 2. Auf richtigen Branch wechseln + Commit holen
git checkout feature/richtiger-branch
git cherry-pick abc1234

# 3. Zurück zum falschen Branch, Commit entfernen
git checkout falscher-branch
git reset --hard HEAD~1

⚠️ Commit bereits gepusht?

  • War der Commit noch nicht gepusht: obiges Vorgehen reicht.
  • War der Commit bereits gepusht: nach dem reset --hard muss force-gepusht werden — aber nur wenn niemand anderes den Branch gepullt hat.
# Nur wenn bereits gepusht und kein anderer den Branch hat
git push --force-with-lease

# War remote schon weiter (diverged)? Erst rebasen
git pull --rebase
# Konflikte lösen falls nötig, dann:
git rebase --continue
