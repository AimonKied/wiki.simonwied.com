42 Inception

Kompletter Einsteigerguide — WordPress in Docker mit NGINX, PHP-FPM und MariaDB.

Docker 42 School NGINX WordPress MariaDB
ℹ️
Für alle, die noch nie von Docker gehört haben und nicht wissen, was hier passiert.
🌐 Projekt

Was macht dieses Projekt?

🗺️
Architektur-Überblick ▶

Dieses Projekt baut eine vollständige WordPress-Website – von Grund auf, mit echten Produktionskomponenten. Alles läuft in Docker-Containern – isolierten Mini-Computern, die auf dem echten Computer laufen.

text · Datenfluss 
Browser  →  NGINX (Port 443, HTTPS)  →  WordPress (PHP)  →  MariaDB (Datenbank)

📋 Ablauf bei einem Seitenaufruf

  • NGINX empfängt die Anfrage (verschlüsselt via HTTPS/SSL)
  • Leitet PHP-Anfragen weiter an WordPress
  • WordPress liest/schreibt Daten in MariaDB (die Datenbank)
  • Die fertige HTML-Seite geht zurück zum Browser
🐳 Docker

Docker Grundlagen

💡
Die Idee in einem Satz ▶

Docker packt ein Programm zusammen mit allem, was es braucht (Betriebssystem-Bibliotheken, Konfiguration, Abhängigkeiten) in eine Box – den Container. Diese Box läuft überall gleich.

🔒
Warum Container statt direkte Installation? ▶

✅ Vorteile

  • Isolation: MariaDB in seinem Container beeinflusst nicht NGINX in seinem
  • Reproduzierbarkeit: Auf jedem Rechner dasselbe Ergebnis
  • Sauberkeit: Kein Müll auf dem Host-System
  • Neustart-Logik: Container können bei Fehler automatisch neu starten
⚙️ Dienste

Die drei Dienste

🚪
NGINX – Der Türsteher ▶

NGINX ist ein Webserver. Er sitzt vorne und nimmt alle Anfragen aus dem Internet entgegen.

📋 Eigenschaften

  • Hört auf Port 443 (HTTPS – verschlüsselt)
  • Hat ein selbst-signiertes SSL-Zertifikat (damit HTTPS funktioniert)
  • PHP-Dateien (*.php) schickt er nicht selbst aus – delegiert an WordPress/PHP-FPM
  • Statische Dateien (Bilder, CSS, JS) liefert er direkt aus
📝
WordPress – Das CMS ▶

WordPress ist das eigentliche Content-Management-System – die Website-Software.

📋 Eigenschaften

  • Läuft nicht als normaler Webserver, sondern als PHP-FPM (FastCGI Process Manager)
  • PHP-FPM hört auf Port 9000 – aber nur intern, nicht von außen erreichbar
  • Beim ersten Start lädt es WordPress herunter, installiert es und legt zwei Benutzer an
  • Speichert alle PHP-Dateien in einem geteilten Volume, das auch NGINX lesen kann
🗄️
MariaDB – Die Datenbank ▶

MariaDB ist eine relationale Datenbank (kompatibel zu MySQL).

📋 Eigenschaften

  • Hört auf Port 3306 – nur intern im Docker-Netzwerk
  • Speichert alle WordPress-Inhalte: Beiträge, Seiten, Benutzer, Einstellungen
  • Beim ersten Start legt es die Datenbank und den Benutzer an, dann startet es normal
⚠️ Regeln

Pflichtregeln & Verbote

🔄
Container müssen bei Absturz neu starten ▶

Pflicht: Jeder Container braucht eine Restart-Policy in docker-compose.yml.

yaml · docker-compose.yml 
services:
  nginx:
    restart: always    # Neustart bei Absturz
  wordpress:
    restart: always
  mariadb:
    restart: always
🚫
Verbotene Entrypoint-Techniken ▶

Ein Container ist keine VM. Hacky Patches um ihn am Laufen zu halten sind verboten:

bash · Verboten ✗ 
tail -f /dev/null              # verboten
sleep infinity                 # verboten
while true; do sleep 1; done  # verboten
bash                           # verboten als Entrypoint

✅ Richtig: Prozess im Vordergrund starten

  • NGINX: exec nginx -g "daemon off;"
  • PHP-FPM: exec php-fpmX.X -F (-F = Foreground)
  • MariaDB: exec mysqld_safe oder exec mysqld
  • Lies: PID 1 best practices für Dockerfiles
🌐
Verbotene Netzwerk-Optionen ▶
yaml · Verboten ✗ 
network: host           # verboten
--link other_container  # verboten
links:                  # verboten

✅ Richtig: Eigenes Docker-Netzwerk

  • Ein networks:-Block muss in docker-compose.yml definiert sein
  • Alle Container hängen im selben selbst definierten Netzwerk (z.B. inception)
👤
WordPress Admin-Benutzername – Einschränkung ▶

Der Admin-Benutzername darf keines dieser Wörter enthalten:

bash · Verboten ✗ 
admin
Admin
administrator
Administrator
admin-123   # oder jede Kombination mit "admin"/"administrator"

✅ Erlaubt

  • user, user, wpmaster – alles ohne "admin" / "administrator" im Namen
  • Im .env: WORDPRESS_ADMIN_USER=user → kein "admin" → ok
🏷️
latest-Tag verboten & Debian/Alpine Version ▶
dockerfile · Verboten ✗ 
FROM debian:latest   # verboten
FROM alpine:latest   # verboten
dockerfile · Richtig ✓ 
FROM debian:bookworm  # konkrete Version
FROM alpine:3.19      # konkrete Version

💡 Welche Version?

  • Pflicht: vorletzte stabile Version von Alpine oder Debian (penultimate stable)
  • Nur Alpine oder Debian – keine anderen Basis-Images erlaubt
  • Keine fertigen Images aus DockerHub (kein FROM wordpress, FROM nginx etc.)
💾
Bind Mounts verboten – nur Named Volumes ▶

Für die zwei persistenten Speicher (WordPress-Dateien und Datenbank) sind ausschließlich Named Volumes erlaubt. Bind Mounts sind verboten.

yaml · Verboten ✗ 
volumes:
  - /home/user/data/wordpress:/var/www/html  # Bind Mount → verboten
  - /home/user/data/mariadb:/var/lib/mysql   # Bind Mount → verboten
yaml · Richtig ✓ 
volumes:
  wp_files:             # Named Volume
  wp_database:          # Named Volume

💡 Datenspeicherort

  • Beide Named Volumes müssen ihre Daten unter /home/login/data auf dem Host speichern
  • Konfigurierbar über driver_opts im volumes:-Block
🔒
Keine Passwörter in Dockerfiles oder Git ▶

❌ Sofortiger Projektdurchfall

  • Credentials, API-Keys oder Passwörter im Git-Repository
  • Passwörter hardcoded in Dockerfiles
  • .env-Datei in Git eingecheckt
  • Secrets-Dateien in Git eingecheckt

✅ Richtig

  • Passwörter in secrets/*.txt (Docker Secrets)
  • secrets/ und srcs/.env in .gitignore
  • Im Dockerfile nur Env-Var-Referenzen, nie Klartextwerte

📄 Dateien

Jede Datei erklärt

🎛️
docker-compose.yml – Die Schaltzentrale ▶

Hier wird definiert, wie alle drei Container gebaut, konfiguriert und verbunden werden.

yaml · docker-compose.yml 
services:
  nginx:       # Dienst 1
  wordpress:   # Dienst 2
  mariadb:     # Dienst 3

networks:
  inception:   # Gemeinsames internes Netzwerk

volumes:
  wp_files:    # Geteilte WordPress-Dateien
  wp_database: # Datenbankdaten

secrets:       # Passwörter aus Dateien lesen

⚠️ Wichtig: depends_on

  • NGINX wartet auf WordPress
  • WordPress wartet auf MariaDB
  • → Container starten in der richtigen Reihenfolge

💾 Volumes auf dem Host

  • wp_files → gemappt auf /home/user/data/wordpress
  • wp_database → gemappt auf /home/user/data/mariadb
  • → Daten überleben Container-Neustarts
🔧
.env und .env.example ▶

Umgebungsvariablen – Konfiguration ohne Passwörter. .env ist in .gitignore – kommt nie in Git.

bash · .env 
DOMAIN_NAME=user.42.fr       # Domain der Website
MYSQL_DATABASE=wordpress      # Name der Datenbank
MYSQL_USER=wp_user            # DB-Benutzer für WordPress
WORDPRESS_TITLE=Inception     # Titel der Website
WORDPRESS_ADMIN_USER=user     # WordPress Admin-Benutzername
WORDPRESS_ADMIN_EMAIL=...     # Admin-E-Mail
WORDPRESS_USER=editor         # Zweiter Benutzer (Rolle: Editor)
WORDPRESS_USER_EMAIL=...      # Editor-E-Mail
🔑
secrets/*.txt – Docker Secrets ▶

Passwörter werden als Docker Secrets verwaltet – nicht als Umgebungsvariablen.

🔐 Warum Secrets statt Env-Vars?

  • Umgebungsvariablen sind im Container-Prozess sichtbar (z.B. env Befehl)
  • Secrets werden als Dateien unter /run/secrets/ eingehängt – sicherer
bash · Passwort lesen 
MYSQL_PASSWORD=$(cat /run/secrets/db_password)
📦
nginx/Dockerfile ▶
dockerfile · nginx/Dockerfile 
FROM debian:bookworm            # Basis: minimales Debian Linux
RUN apt-get install nginx openssl  # Installiere NGINX und SSL-Tools
COPY conf/nginx.conf ...        # Kopiere Konfiguration ins Image
COPY tools/setup-nginx.sh ...   # Kopiere Startskript
EXPOSE 443                      # Dokumentiert: dieser Container nutzt Port 443
ENTRYPOINT ["/usr/local/bin/setup-nginx.sh"]  # Dieses Skript startet beim Container-Start
🔐
nginx/setup-nginx.sh ▶
bash · setup-nginx.sh 
# Generiert ein selbst-signiertes SSL-Zertifikat (beim ersten Start)
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
    -keyout nginx.key -out nginx.crt \
    -subj "/CN=${DOMAIN_NAME}"

# Startet NGINX im Vordergrund (wichtig für Docker!)
exec nginx -g "daemon off;"

💡 Warum daemon off?

  • Docker erwartet einen Prozess im Vordergrund
  • Würde NGINX im Hintergrund laufen, denkt Docker: Container fertig → stoppt ihn
⚙️
nginx/nginx.conf ▶
nginx · nginx.conf 
server {
    listen 443 ssl;                    # Nur HTTPS, kein HTTP
    server_name user.42.fr;

    ssl_protocols TLSv1.2 TLSv1.3;    # Nur sichere SSL-Versionen

    root /var/www/html/wordpress;      # Hier liegen die WP-Dateien

    location ~ \.php$ {
        fastcgi_pass wordpress:9000;   # PHP → weiter an WordPress-Container
    }
}

💡 Wie funktioniert wordpress:9000?

  • Beide Container sind im selben Docker-Netzwerk inception
  • Docker löst den Hostnamen wordpress automatisch auf die Container-IP auf
📦
wordpress/Dockerfile ▶

Installiert alle benötigten PHP-Pakete und richtet PHP-FPM für TCP-Port 9000 ein.

📦 Installierte Pakete

  • php8.2-fpm – PHP-Prozessmanager
  • php8.2-mysql – PHP kann mit MySQL/MariaDB reden
  • php8.2-gd, mbstring, xml, zip – von WordPress benötigt
  • wp-cli – Kommandozeilen-Tool um WordPress zu installieren/konfigurieren

💡 Unix-Socket → TCP Port 9000

  • Standard PHP-FPM nutzt Unix-Socket (lokale Datei)
  • Zwischen Containern funktioniert nur TCP → daher Port 9000
🚀
wordpress/wp-setup.sh ▶

Beim ersten Start (wenn wp-config.php noch nicht existiert):

📋 Ablauf

  • 1. Wartet bis MariaDB erreichbar ist (Port 3306)
  • 2. Lädt WordPress herunter via wp core download
  • 3. Erstellt wp-config.php mit Datenbankverbindungsdaten
  • 4. Installiert WordPress (legt Admin-Account an)
  • 5. Erstellt zweiten Benutzer mit Editor-Rolle
  • 6. Setzt Dateiberechtigungen auf www-data

⚡ Folgestart

  • wp-config.php existiert bereits → überspringt alles → startet direkt PHP-FPM
🗄️
mariadb/50-server.cnf ▶
ini · 50-server.cnf 
bind-address = 0.0.0.0   # Hört auf alle Netzwerkinterfaces (nicht nur localhost)
port = 3306
datadir = /var/lib/mysql  # Wo Daten gespeichert werden

💡 Warum bind-address = 0.0.0.0?

  • Standard: MariaDB hört nur auf localhost (127.0.0.1)
  • WordPress läuft in einem anderen Container → andere IP
  • 0.0.0.0 → hört auf allen Interfaces → WordPress kann sich verbinden
🔄
mariadb/init_db.sh ▶

Beim ersten Start (Datenverzeichnis leer):

📋 Ablauf

  • 1. Initialisiert das MySQL-Datenverzeichnis (mysql_install_db)
  • 2. Startet temporären MariaDB-Server ohne Netzwerk (nur für Setup)
  • 3. Führt Setup-SQL aus: root-Passwort setzen, DB erstellen, User erstellen, Rechte vergeben
  • 4. Fährt temporären Server herunter
  • 5. Startet MariaDB normal mit Netzwerk

Datenfluss bei einem Seitenaufruf ▶

📋 Schritt für Schritt

  • 1. Browser → https://user.42.fr → NGINX Port 443
  • 2. NGINX prüft: ist es eine .php-Datei? → Ja
  • 3. NGINX schickt Anfrage an wordpress:9000 (FastCGI)
  • 4. PHP-FPM verarbeitet die PHP-Datei
  • 5. WordPress-Code fragt MariaDB: "Gib mir Beitrag #5"
  • 6. MariaDB antwortet mit den Daten
  • 7. WordPress baut HTML zusammen
  • 8. HTML geht zurück: WordPress → NGINX → Browser

📦
"Was ist der Unterschied zwischen Image und Container?" ▶

💡 Erklärung

  • Image = gespeicherter Bauplan (auf Festplatte)
  • Container = laufende Instanz davon (im RAM)
  • Aus einem Image können viele Container starten
🔒
"Warum kein HTTP, nur HTTPS?" ▶

Das ist Anforderung des Projekts. NGINX hört nur auf Port 443 (HTTPS), nicht auf Port 80 (HTTP).

💾
"Wo werden WordPress-Dateien gespeichert?" ▶

Im Volume wp_files, das auf /home/user/data/wordpress auf dem Host zeigt. Auch wenn der Container gelöscht wird, bleiben die Daten dort erhalten.

💾
"Was ist der Unterschied zwischen Bind Mount und Named Volume?" ▶
Bind MountNamed Volume
DefinitionDirekter Pfad vom Host ins ContainerDocker verwaltet den Speicherort
Syntax/host/pfad:/container/pfadvolume_name:/container/pfad
SpeicherortBeliebiger Host-Pfad/var/lib/docker/volumes/… (oder custom)
In InceptionVerbotenPflicht

🔍 Wie funktioniert ein Bind Mount?

  • Du gibst einen konkreten Pfad auf dem Host an – z.B. /home/user/data/wordpress
  • Der Kernel hängt diesen Ordner direkt in den Container ein (Linux-mount --bind)
  • Container und Host sehen dieselben Dateien in Echtzeit – kein Kopieren
  • Der Pfad muss auf dem Host existieren, sonst legt Docker ihn als leeren Ordner an
yaml · Bind Mount Beispiel 
services:
  wordpress:
    volumes:
      - /home/user/data/wordpress:/var/www/html  # ← Bind Mount

🔍 Wie funktioniert ein Named Volume?

  • Docker legt einen eigenen Speicherbereich an – standardmäßig unter /var/lib/docker/volumes/<name>/_data
  • Im volumes:-Block des Compose-Files wird das Volume deklariert
  • Mit driver_opts kann der Host-Pfad auf /home/login/data/… gesetzt werden (Subject-Pflicht)
  • Docker kennt das Volume: docker volume ls, docker volume inspect
  • Bleibt erhalten auch wenn alle Container gelöscht werden – erst docker volume rm löscht es
yaml · Named Volume mit custom Pfad 
services:
  wordpress:
    volumes:
      - wp_files:/var/www/html         # ← Named Volume

volumes:
  wp_files:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: /home/user/data/wordpress  # Speicherort auf Host

💡 Warum Named Volumes in Inception?

  • Named Volumes sind portabler – Docker kennt und verwaltet sie (docker volume ls)
  • Bind Mounts hängen direkt am Host-Dateisystem – weniger isoliert
  • Subject-Pflicht: Daten unter /home/login/data via driver_opts konfiguriert
🔑
"Warum Passwörter in .txt-Dateien statt in .env?" ▶

💡 Erklärung

  • Docker Secrets sind sicherer – werden als gemountete Dateien unter /run/secrets/ eingebunden
